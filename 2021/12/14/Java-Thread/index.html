线程

Some Basic Concepts

线程是独立的线程，他代表独立的执行空间。

Thread是Java中用来表示线程的类，要建立线程就得创建Thread。

每一个Java应用程序会启动一个主线程—将main( )放在自己执行空间的最开始出。Java虚拟机会负责主线程的启动(如垃圾回收所需的系统用线程)。程序员要负责启动自己建立的线程。

What does having more than one executing space symbolize?

当有超过一个以上的执行空间时，看起来好像是有好几件事同时发生。但实际上，执行动作可以在执行空间中快速地来回切换。Java也只是个在底层操作系统上执行的进程。一旦轮到Java执行时，Java虚拟机会执行目前执行空间最上面的。

线程要记录的一项事务是目前线程执行空间做到哪里。

举个栗子：

Java虚拟机调用main( )主线程。

main( )中建立了一个Thread对象t，并启动了新线程t.start( )。则此时，堆上新增线程t.start( )，另起一执行空间，启动自建线程中的run( )，在新线程启动期间，main( )会暂停执行。

之后，会在主线程和用户自建线程之间来回切换，直到二者全部执行完毕。

如何启动新的线程？

  1. 建立Runnable对象(线程的任务)

    Runnable threadJob = new MyRunnable();

  2. 建立Thread对象(执行工人)并赋值Runnable(任务)

    Thread myThread = new Thread(threadJob);

    把Runnable对象传给Thread的构造函数。这会告诉Thread对象要把那个方法放在执行空间区运行—Runnable的run( )方法。

  3. 启动Thread

    myThread.start( );

    在还没有调用Thread的start( )方法之前，什么也不会发生。当新的线程启动之后，它会把Runnable对象的方法摆到新的执行空间中。

对于Thread而言，它相当于一个工人，对于Runnable而言，它相当于这个工人要完成的工作。Runnable带有会放在执行空间的第一项的方法：run( )。

Thread对象需要任务，任务是线程在启动时去执行的工作。run( )该任务是新线程空间上的第一个方法，且一点更要长得像下面这样:

public void run( ) {

​ //会被新线程执行的代码

}

Runnable定义了一个协约。因为Runnable是个接口，线程的任务可以被定义在任何实现Runnable的类上。线程只在乎传入给Thread的构造函数的参数是否为实现Runnable的类。当把Runnable传给Thread的构造函数时，实际上就是在给Thread取得run( )的方法，即给了Thread一项任务。Runnable接口只有run( )这一个方法。

在调用start( )开始执行线程之前，对象只是个Thread实例，并不是真正的线程。

新建线程的三个状态

  1. 新建

    Thread t = new Thread(r);

Thread的实例已经创建，但还没有启动。也就是说，有Thread对象，没有执行中的线程。

  1. 可执行

    t.start( );

当启动线程时，它会变成可执行的状态。这时，线程已经布置好执行空间。

  1. 执行中(线程的目标)

    轮到它执行

所有的线程都在等待这一刻，成为正在执行中的那一个。这只能靠Java虚拟机的线程调度机制来决定。有时也能对Java虚拟机选择执行的线程给点意见，但无法强迫它把线程从可执行状态移动到执行中。

一旦线程进入可执行状态，他会在可执行与执行中两种状态中来来去去，同时也有另一种状态：暂时不可执行(又称为被堵塞状态)。

典型的可执行/执行中循环：

通常线程会在可执行与执行中两种状态中来回交替。因为Java虚拟机的线程调度会把线程跳出来运行又把它踢回去以让其他线程有执行机会。

线程有可能会暂时被挡住

调度器会因为某些原因把线程送进去关一阵子。例如线程执行到等待Socket输入串流的程序段，但没有数据可供读取。调度器会把线程移出可执行状态等。会导致线程暂时失能。

无法掌控的线程调度器

无法控制调度，没有API可以可调用调度器。调度器在不同的虚拟机上有不同的行为。最重要的是，调度无法确定。

也就是说，同一个多线程的程序，在不同的主机上运行，会导致不同的结果。甚至于同一个程序在同一台主机上运行的结果也是不确定的。不能让程序依靠调度的特定行为来保持执行的正确性！

吃个栗子：

调度不能保证任何的执行时间和顺序，所以你不能期待他会完全地平均分配执行，最多只能影响sleep的最小保证时间。

想要确保其他线程有机会执行的话，就把线程放进睡眠状态。

锁

Ryan和Monica的透支银行余额事件

描述

解决方案

每个对象都有个锁。大部分时间都没有锁上。对象的锁只会在有同步化的方法上起作用。当对象有一个或多个同步化的方法时，线程只有在取得对象锁的钥匙时，才能进入同步化的方法。

锁不是配在方法上的，而是配在对象上。如果对象有两个同步化的方法，就表示两个线程无法进入同一个方法，也表示两个线程无法进入不同的方法。如果有多个方法可能会操作对象的实例变量，则这些方法都应该有同步化的保护。

同步化的目标是要保护重要的数据。但要注意，锁住的不是数据，而是存取数据的方法。

所以线程在开始执行并遇到同步化的方法时，会认知到他需要对象的钥匙才能进入方法。他会取得钥匙，这是由Java虚拟机来处理的，没有可以存取对象锁的API。那么此时，他会保护好这把钥匙，直到完成同步化方法。与此同时，其他线程不可以进入该对象的同步化方法，因为只有这一把钥匙。

只有当对象具有同步化方法时，才会上锁。而且线程只有取得钥匙后，才能进入该方法。也就是说，线程只有在没有其他线程已经进入的情况下，才能进入。

“丢失更新”问题

描述

这是另一个典型的并行性(concurrency)问题。丢失更新(lost update)有一种特定的过程。

吃个栗子：

  1. 取得账户余额 int i = balance;
  2. 将账户余额增加 balance = i + 1;

正常情况下，都是直接balance++; 但怎么说呢，偶尔会因为表达式的复杂，只得使用以上方法。

看上去，这样没什么问题。

那么，若此时有两个线程A和B在执行这个方法。

A：取余额0，增加1，变为1；

​ 再取余额1，增加1，变为2；

B：取余额2，增加1，变为3；

​ 取余额3，B睡了，此时未增加。

A：取余额3，增加1，变为4；

​ 取余额4，增加1，变为5。

B：苏醒，将其的i=3，增加1，余额balance变为4…….

很明显，B更改了A增加的值，丢失了A进行的更新。

此时，就需要同步化机制来帮忙，将 int i = balance; balance = i + 1;锁成整体，变得不可分割。

死锁现象

死锁会发生是因为两个线程互相持有对方正在等待的东西。没有方法可以脱离这个情况。所以只好一直等，一直等。

Java中没有处理死锁的机制，甚至它都不知道死锁的发生，所以要小心设计程序。如果经常编写多线程的程序，去看看O’Reilly出版的“Java Thread”。

总结

  1. Thread.sleep()这个静态方法可以强制线程进入等待状态到过了设定的时间为止。
  2. sleep()方法可能会抛出InterruptedException异常，所以要包在try/catch块中，或者把他也声明出来。
  3. 可以用setName来给线程命名，通常是用来除错的。
  4. 如果两个或以上线程存取堆上相同的对象可能会出现严重的问题。
  5. 如果两个或以上的线程存取相同的对象可能会引发数据的损毁。
  6. 使用synchronized这个关键词修饰符，可以防止两个线程同时进入同一个对象的同一个方法。
  7. 对象就算是有多个同步化的方法，也还是只有一个锁。一旦某个线程进入该对象的同步化方法，其他线程就无法进入该对象上的任何同步化方法。
  8. 不只对象有锁，类本身也是有锁的。也就是说，假如有3个Dog对象，那么此时应该是有4个锁。
  9. 故而，对静态变量进行同步化，线程想要进入方法就需要先获得类的锁。
  10. 同步化的根本目的是为了防止步骤被分割，也就是指定某段工作要在不能分割的状态下执行。单独的操作并不重要，重要的是拥有多个步骤的一个方法。
